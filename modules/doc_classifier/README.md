# doc_classifier

## Назначение
Автономный модуль AI-распознавания и классификации рабочей документации.

Модуль:
- берет файлы из реестра `ingest_registry`;
- отправляет их во внешний AI через настройки из корневого `.env`;
- распознает и классифицирует тип рабочей документации;
- сохраняет отдельно результат распознавания и результат классификации;
- хранит структурированные поля, пригодные для последующей сборки исполнительной документации;
- показывает статусы, типы документов и извлеченные данные в собственном UI.

Модуль не делает:
- импорт из внешних источников;
- ведение первичного реестра файлов;
- проектные связи и межмодульную аналитику.

## Структура папки
- `main.py` - точка входа модуля.
- `AGENTS.md` - локальная политика агента для модуля.
- `README.md` - описание модуля и карта файлов.
- `requirements.txt` - локальные Python-зависимости модуля.
- `app/` - прикладная логика модуля.
- `db/` - локальная SQLite БД, схема и репозиторий.
- `ui/` - локальный HTTP UI.
- `data/` - каталог для локальной БД модуля.
- `tests/` - локальные проверки UI-статусов и служебной логики модуля.

## Карта файлов
- `main.py` - запускает bootstrap и HTTP UI.
- `requirements.txt` - зависимости `requests`, `pypdf`, `openpyxl` для AI-обработки и разбора файлов.
- `app/config.py` - локальная конфигурация модуля, пути к БД `doc_classifier` и `ingest_registry`, настройки внешнего AI.
- `app/bootstrap.py` - старт логирования, инициализация БД, синхронизация реестра и запуск сервера.
- `app/ai_client.py` - HTTP клиент внешнего AI и каталог типов рабочей документации с целевыми полями.
- `app/file_readers.py` - чтение текстовых, офисных и PDF файлов; разбиение PDF на части до 10 МБ.
- `app/models.py` - модель записи документа.
- `app/pipeline.py` - AI-пайплайн распознавания и классификации документа.
- `app/registry_source.py` - чтение файлов из SQLite-реестра `ingest_registry` в режиме только чтения.
- `app/service.py` - прикладной сервис для dashboard, синхронизации реестра и повторного прогона с учетом `cabinet_id`.
- `db/setup.py` - создание таблиц `documents` и `module_events`.
- `db/repository.py` - доступ к SQLite, синхронизация записей из реестра и обновление результатов.
- `ui/server.py` - HTTP сервер и маршруты UI.
- `ui/templates.py` - HTML шаблоны страниц.
- `tests/test_cabinet_id.py` - минимальные проверки `cabinet_id` для repository и UI-сценариев `sync/rerun`.
- `tests/test_ui_statuses.py` - проверки, что fallback-документы помечаются как требующие проверки и не маскируются под успешно распознанные.
- `data/doc_classifier.sqlite3` - локальная БД модуля, создается при первом запуске.
- `data/tmp/` - временные части PDF при разбиении больших файлов.

## Как запускать
Команда:

```bash
pip install -r modules/doc_classifier/requirements.txt
./modulectl start doc_classifier
```

Проверка статуса и доступности:

```bash
./modulectl status doc_classifier
./modulectl health doc_classifier
```

Внешний запуск по умолчанию:
- модуль слушает `0.0.0.0:8002`, если `DOC_CLASSIFIER_HOST` не задан отдельно;
- для доступа с других машин порт нужно открыть один раз: `sudo ufw allow 8002/tcp`.

Требуемые переменные окружения из корневого `.env`:
- `OPENAI_API_KEY`
- `OPENAI_MODEL`
- `OPENAI_MAX_UNITS_PER_REQUEST`
- `OPENAI_MAX_INPUT_CHARS_PER_REQUEST`

Адрес UI:
- локально: `http://127.0.0.1:8002`
- извне: `http://<server-ip>:8002`

Порт:
- `8002`

## UI
Доступны страницы:
- `/` - страница статуса модуля, агрегированные счетчики и большой блок `Рабочая документация`.
- `/documents` - список документов со статусами.
- `/documents/<id>` - карточка документа.

Основной сценарий интерфейса:
- desktop-first для широких экранов и рабочих реестров;
- summary сверху, ниже фильтры, сортировка и действия;
- основные списки и группировки выводятся таблицами со sticky-заголовками;
- поддерживаются режимы плотности `Обычно` и `Компактно`.
- сценарии `sync` и `rerun` сохраняют текущий `cabinet_id` в относительных ссылках, формах и редиректах.
- для shell-прокси формы используют обычный `action`, чтобы cabinet-aware POST-маршруты корректно доходили до модуля и не ломались на переписывании ссылок.

На карточке документа показываются:
- исходный статус;
- статус распознавания;
- статус классификации;
- распознанный текст;
- структурированные извлеченные поля;
- технические признаки обработки;
- результат классификации;
- кнопка повторного прогона документа.

На главной странице после верхних карточек выводится:
- блок `Рабочая документация`;
- подблоки по типам документов;
- внутри каждого подблока табличный список: материал, статус, распознанные данные, дата обновления и ссылка на исходный файл.

В реестре `/documents`:
- основной столбец показывает название материала или продукции, если оно распознано;
- имя исходного файла показывается второй строкой под названием материала;
- каждое распознанное поле выводится отдельной колонкой, а не общим блоком;
- кнопка `Файл` открывает конкретный исходный документ на Google Drive или Яндекс Диске в отдельном окне, а не папку источника.

Пользовательские статусы в списке документов:
- `Готово` - есть подтвержденные распознанные данные и завершенная классификация.
- `Требует проверки` - тип назначен предварительно, но подтвержденного распознавания текста нет.
- `Не распознан` - полезные данные не извлечены и тип не подтвержден.
- `Ошибка` - обработка завершилась с ошибкой.

## База данных
У модуля своя локальная SQLite БД:
- путь: `data/doc_classifier.sqlite3`

Основные таблицы:
- `documents` - документы из `ingest_registry`, их исходный статус, статус распознавания, статус классификации, распознанный текст, структурированные поля и результаты классификации; все записи относятся к конкретному `cabinet_id`.
- `module_events` - локальный журнал событий модуля по выбранному `cabinet_id`.

Изоляция кабинетов:
- синхронизация ingest-документов удаляет и обновляет записи только внутри текущего `cabinet_id`;
- список документов, карточка документа, `sync` и `rerun` не смешивают данные между кабинетами;
- для проверки этого поведения есть отдельные минимальные тесты repository и UI.
- ошибка обработки одного файла не должна обрывать cabinet-sync целиком: остальные документы того же кабинета продолжают обрабатываться.

Статусы разделены на три независимых поля:
- `source_status`
- `recognition_status`
- `classification_status`

## Основные сценарии
1. Запустить модуль через `./modulectl start doc_classifier`.
2. При первом внешнем запуске открыть порт `8002/tcp` в `ufw`.
3. Убедиться, что в корневом `.env` заданы переменные внешнего AI.
4. Открыть UI на порту `8002`.
5. При многокабинетной работе открывать UI с `cabinet_id`, например `http://127.0.0.1:8002/?cabinet_id=default`.
6. Нажать `Синхронизировать реестр и распознать документы` либо дождаться обработки на старте.
7. Посмотреть общий статус модуля на главной странице.
8. Открыть блок `Рабочая документация` и проверить подблоки по типам документов.
9. Открыть список документов.
10. Перейти в карточку документа и посмотреть результат распознавания и классификации.
11. Повторно прогнать документ без повреждения существующей структуры данных.

## Примечания по стартовой реализации
- Реестр читается из `modules/ingest_registry/data/ingest_registry.sqlite3` только в режиме чтения.
- Для PDF больше 10 МБ модуль делит файл на части по страницам так, чтобы каждая часть помещалась в лимит 10 МБ.
- Для PDF модуль сначала пытается отправить сам файл во внешний AI с локальным text hint, и только при ошибке file-input уходит в text/image fallback или в классификацию по имени файла.
- CSV и Excel отправляются во внешний AI как текст, потому что API не принимает эти форматы как файлы.
- Типы документов и целевые поля ориентированы на рабочую и исполнительную документацию: документы качества, проекты, сметы, приказы, журналы, акты, исполнительные схемы, протоколы испытаний и разрешительные документы.
