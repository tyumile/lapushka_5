# ingest_registry

## Назначение
Автономный модуль импорта и синхронизации файлов из публичных папок Google Drive, Yandex Disk и локальной загрузки.

Стартовая версия модуля уже умеет:
- запускать свой локальный HTTP UI;
- принимать локальную загрузку файлов;
- принимать публичные ссылки на папки Google Drive и Yandex Disk;
- синхронизировать файлы из добавленных источников;
- хранить реестр файлов в собственной SQLite БД;
- показывать список источников и итоги последней синхронизации;
- хранить журнал действий модуля.

Стартовая версия модуля пока не умеет:
- делать OCR;
- делать классификацию;
- выполнять проектную аналитику.

## Порт
- `8001`

## Запуск
Из корня проекта:

```bash
./modulectl start ingest_registry
```

Проверка статуса и доступности:

```bash
./modulectl status ingest_registry
./modulectl health ingest_registry
```

После запуска UI доступен по адресу:

```text
http://127.0.0.1:8001/
```

Для доступа с другой машины используйте:

```text
http://<server-ip>:8001/
```

## Структура папки
- `main.py` — точка входа, поднимает локальный HTTP сервер модуля.
- `AGENTS.md` — локальная политика агента для `ingest_registry`.
- `README.md` — описание модуля и карта файлов.
- `app/` — app-слой: конфигурация, bootstrap, сервисная логика.
- `db/` — db-слой: схема SQLite и repository для реестра.
- `ui/` — ui-слой: HTTP handler и HTML страницы состояния.
- `data/` — локальные данные модуля, SQLite БД и сохраненные файлы.
- `logs/` — локальные логи запуска и действий.
- `tests/` — тесты cabinet-aware HTTP, repository migration и service-сценариев синхронизации.
- `prompts/` — директория под локальные промпты, если понадобятся.

## Карта файлов
- `main.py` — entrypoint.
- `app/config.py` — настройки путей, имени модуля и порта.
- `app/bootstrap.py` — инициализация логирования, БД, сервиса и HTTP сервера.
- `app/connectors.py` — клиенты для локального хранилища, публичного Google Drive и публичного Yandex Disk.
- `app/service.py` — бизнес-логика модуля: bootstrap, загрузка файла, управление источниками, синхронизация, dashboard data.
- `db/schema.py` — SQL схема таблиц `sources`, `file_registry`, `activity_log`.
- `db/repository.py` — доступ к SQLite БД и операции с реестром.
- `ui/http.py` — обработка `GET /`, `POST /upload`, `POST /sources/add`, `POST /sync/<source_id>`, сохранение `cabinet_id` в query и рендер HTML.
- `data/ingest_registry.sqlite3` — собственная БД модуля.
- `data/storage/bootstrap/welcome.txt` — bootstrap-файл, который подтверждает работу начального реестра.
- `data/storage/uploads/` — каталог локально загруженных файлов.
- `logs/ingest_registry.log` — файл логов модуля.

## UI
Страница состояния сделана как desktop-first экран для регулярной работы с реестром:
- сверху короткий summary с ключевыми показателями по файлам, источникам и проблемным записям;
- ниже блок быстрых действий: добавление Google Drive, Яндекс Диска и локальная загрузка;
- ниже панель фильтров и режимов: отбор по типу источника, порядок строк и плотность `Обычно` / `Компактно`;
- далее табличный реестр источников с состоянием синхронизации, статистикой последнего запуска и следующим шагом для проблемных строк;
- основной рабочий блок — широкая таблица реестра файлов с закрепленной первой колонкой, читаемыми статусами, деталями по клику и удобным сравнением строк;
- внизу — табличный журнал последних действий модуля.

Cabinet-aware поведение:
- активный кабинет задается через `?cabinet_id=<значение>`;
- все фильтры, ссылки сортировки и режимы плотности сохраняют текущий `cabinet_id`;
- формы `Добавить Google Drive`, `Добавить Яндекс Диск`, `Загрузить файл` и кнопки `Синхронизировать` отправляют `POST` в тот же кабинет;
- после `POST` редирект возвращает пользователя в тот же кабинет, а не в общий `default`.
- локальные загрузки физически раскладываются по каталогам `data/storage/uploads/<cabinet_id>/...`, а синхронизация `local_upload` читает только каталог текущего кабинета.

## База данных
У модуля своя отдельная SQLite БД:

```text
data/ingest_registry.sqlite3
```

Таблицы:
- `sources` — список источников, ссылка, дата добавления, статус и итог последней синхронизации; каждая запись привязана к `cabinet_id`.
- `file_registry` — реестр файлов, привязанный к источнику по `source_id`, с идентификатором файла, путем, именем, размером, числом листов, датой появления и датой изменения; все записи фильтруются по `cabinet_id`.
- `activity_log` — журнал ключевых действий модуля в рамках выбранного `cabinet_id`.

## Основные сценарии
1. Запустить модуль через `./modulectl start ingest_registry`.
2. Открыть UI на `http://127.0.0.1:8001/` или `http://<server-ip>:8001/`.
3. При многокабинетной работе открыть нужный кабинет через query, например `http://127.0.0.1:8001/?cabinet_id=default` или `http://127.0.0.1:8001/?cabinet_id=test`.
4. Добавить ссылку на публичную папку Google Drive или Yandex Disk.
5. Нажать `Синхронизировать` у нужного источника.
6. Увидеть обновленный реестр файлов и статистику последней синхронизации только для текущего `cabinet_id`.
7. При необходимости загрузить локальный файл через форму upload; запись попадет только в текущий кабинет.

- Есть разделение на `app`, `db`, `ui`.
- Есть отдельная БД модуля.

- Есть работающие публичные источники Google Drive и Yandex Disk.

## Ограничения стартовой версии
- Для Google Drive используется публичный просмотр папки без OAuth, поэтому часть метаданных может быть недоступна для отдельных файлов.
- Количество листов определяется автоматически в первую очередь для PDF-файлов; для остальных форматов поле может быть пустым.
- Внешние вызовы идут с таймаутами, но зависят от доступности публичных ссылок и сети.

## Проверка
