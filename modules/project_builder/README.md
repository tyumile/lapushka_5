# project_builder

## Назначение
Автономный модуль формирования проектов, сопоставления документов и определения нехватки материалов.

Стартовая версия модуля уже умеет:
- запускать свой локальный HTTP UI;
- хранить проекты, документы, связи и нехватки в собственной SQLite БД;
- показывать список проектов с раскрытием деталей прямо в таблице и отдельную fallback-страницу проекта;
- объяснять, почему документ отнесен к проекту;
- отдельно хранить найденные связи и дефициты;
- фиксировать локальные записи о действиях модуля и стартовую запись в журнале реестра задач.

Стартовая версия модуля не умеет:
- импортировать файлы;
- выполнять OCR;
- выполнять первичную классификацию;
- читать или изменять данные других модулей.

## Порт
- `8003`

## Запуск
Из корня проекта:

```bash
./modulectl start project_builder
```

Проверка статуса и доступности:

```bash
./modulectl status project_builder
./modulectl health project_builder
```

После запуска UI доступен по адресу:

```text
http://127.0.0.1:8003/
```

## UI
Доступны страницы:
- `/` — статус модуля, счетчики, краткий список проектов и локальный журнал реестра задач.
- `/projects` — основной рабочий экран со списком всех проектов и раскрытием деталей прямо под строкой таблицы.
- `/projects/<id>` — техническая fallback-страница проекта.

Во встроенных деталях проекта и на fallback-странице показываются:
- список документов проекта;
- объяснение, почему документ отнесен к проекту;
- найденные связи между документами;
- список нехваток с критичностью;
- агрегированная полнота проекта.

Cabinet-aware режим:
- все UI-маршруты поддерживают `cabinet_id` через query, например `/projects?cabinet_id=default`;
- список проектов, встроенные детали, fallback-страница проекта и AI-анализ читают только данные текущего кабинета;
- результаты анализа сохраняются в отдельные каталоги вида `data/project_analyses/<timestamp>_cabinet_<cabinet_id>_project_<id>/`;
- в каждом `project_analysis.json` хранится top-level поле `cabinet_id`, а `AnalysisStore` фильтрует результаты по текущему кабинету и не смешивает соседние кабинеты.

## База данных
У модуля своя отдельная SQLite БД:

```text
data/project_builder.sqlite3
```

Таблицы:
- `projects` — проекты и общий статус полноты по `cabinet_id`.
- `project_documents` — документы проекта и причина сопоставления по `cabinet_id`.
- `project_links` — найденные связи между документами по `cabinet_id`.
- `project_deficits` — отсутствующие обязательные материалы по `cabinet_id`.
- `module_events` — журнал ключевых событий модуля по `cabinet_id`.
- `task_registry_log` — локальный журнал записей в формате SQL-реестра задач по `cabinet_id`.

## Основные сценарии
1. Запустить модуль через `./modulectl start project_builder`.
2. Открыть UI на `http://127.0.0.1:8003/`.
3. При многокабинетной работе передавать `cabinet_id` через query, например `http://127.0.0.1:8003/?cabinet_id=default`.
4. Посмотреть список проектов и общие счетчики.
5. На статусной странице нажать кнопку `Анализ проекта`, чтобы модуль запустил внешний анализ по наиболее вероятному проектному файлу текущего кабинета.
6. На экране `/projects` раскрыть нужный проект прямо в строке таблицы и увидеть документы, связи, нехватки и разбивку по секциям без перехода между страницами.
7. При необходимости запустить тот же анализ вручную через CLI:

```bash
pip install -r modules/project_builder/requirements.txt
venv/bin/python modules/project_builder/scripts/run_project_analysis.py --cabinet-id <cabinet_id> --project-document-id <file_registry_id> --source-id <source_id>
```

Важно:
- загрузка файлов в `ingest_registry` не создает проекты в `project_builder` автоматически;
- обработка документов в `doc_classifier` тоже не создает проекты автоматически, а только обогащает метаданные для сопоставления;
- чтобы кабинет появился в UI `project_builder`, нужен отдельный запуск проектного AI-анализа; теперь его можно стартовать кнопкой `Анализ проекта` в UI или вручную через `scripts/run_project_analysis.py --cabinet-id <cabinet_id> ...`, который сохранит `project_analysis.json` для этого кабинета.

6. Проверить, что в `modules/project_builder/data/project_analyses/<run_id>/` созданы:
   - `request_context.json` — полный payload проекта и списка документов из `ingest_registry`;
   - `project_analysis.json` — строгий JSON для будущей итоговой страницы проекта с top-level полем `cabinet_id`.

Для безопасной проверки cabinet-aware режима можно использовать dry-run:

```bash
python3 modules/project_builder/scripts/run_project_analysis.py --cabinet-id default --project-document-id <file_registry_id> --dry-run
```

Dry-run не вызывает внешний AI, но создает раздельные артефакты по каждому кабинету и позволяет проверить, что результаты не смешиваются.

## Внешний AI-анализ проекта
Скрипт `scripts/run_project_analysis.py`:
- читает проектный документ и список документов из `modules/ingest_registry/data/ingest_registry.sqlite3` только в режиме read-only;
- берет настройки внешнего агента из корневого `.env`;
- отправляет файл проекта и JSON-контекст во внешнего агента с промптом `Инженер ПТО`;
- сохраняет результат строго в `project_analysis.json`.

Поддерживаемые аргументы:
- `--cabinet-id` — cabinet-контекст для чтения `ingest_registry`, `doc_classifier` и записи `project_analysis.json`;
- `--project-document-id` — обязательный `file_registry.id` документа проекта;
- `--project-file-path` — локальный путь к файлу проекта, если в `ingest_registry` хранится только удаленная карточка без скачанного файла;
- `--source-id` — ограничить список документов конкретным источником;
- `--document-id` — ограничить список документов конкретными id;
- `--dry-run` — собрать payload без вызова внешнего агента.

Структура итогового JSON уже соответствует будущему UI:
- верхний блок `project`;
- массив `modules`;
- внутри каждого модуля массивы `materials` и `norms`;
- у материалов есть имя, документ, номер, дата, ссылка на файл на диске и ссылка на страницу проекта, где материал указан;
- у норм есть точное название нормы и источник в проекте.

## Что подготовлено в стартовой реализации
- Есть entrypoint через `main.py`.
- Есть разделение на `app`, `db`, `ui`.
- Есть собственная SQLite БД модуля.
- Есть seeded демонстрационные проекты без импорта и OCR.
- Есть локальный UI со списком проектов, связями и нехватками.
- Есть локальный лог и таблица `task_registry_log` для фиксации действий в формате реестра задач.

## Ограничения стартовой версии
- Данные пока только демонстрационные и seeded при первом запуске.
- Полнота проекта считается по простому правилу: заполненные документы против открытых дефицитов.
- Централизованная фиксация задач должна идти через общий `./task add ... --sync`; локальная таблица `task_registry_log` используется только как внутренний журнал модуля.
