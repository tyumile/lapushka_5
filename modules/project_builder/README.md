# project_builder

## Назначение
Автономный модуль формирования проектов, сопоставления документов и определения нехватки материалов.

Стартовая версия модуля уже умеет:
- запускать свой локальный HTTP UI;
- хранить проекты, документы, связи и нехватки в собственной SQLite БД;
- показывать список проектов и карточку проекта;
- объяснять, почему документ отнесен к проекту;
- отдельно хранить найденные связи и дефициты;
- фиксировать локальные записи о действиях модуля и стартовую запись в журнале реестра задач.

Стартовая версия модуля не умеет:
- импортировать файлы;
- выполнять OCR;
- выполнять первичную классификацию;
- читать или изменять данные других модулей.

## Порт
- `8103`

## Запуск
Из корня проекта:

```bash
./modulectl start project_builder
```

Проверка статуса и доступности:

```bash
./modulectl status project_builder
./modulectl health project_builder
```

После запуска UI доступен по адресу:

```text
http://127.0.0.1:8103/
```

## Структура папки
- `main.py` — точка входа модуля.
- `AGENTS.md` — локальная политика агента для `project_builder`.
- `README.md` — описание модуля и карта файлов.
- `app/` — app-слой: конфигурация, bootstrap, сервис и модели.
- `db/` — db-слой: схема SQLite и repository.
- `ui/` — ui-слой: HTTP сервер и HTML шаблоны.
- `data/` — локальные данные модуля и SQLite БД.
- `logs/` — локальные логи модуля.
- `tests/` — директория под будущие тесты.
- `prompts/` — директория под будущие локальные промпты, если понадобятся.

## Карта файлов
- `main.py` — entrypoint, запускает bootstrap.
- `app/config.py` — настройки модуля, пути, порт и имя сервиса.
- `app/bootstrap.py` — старт логирования, инициализация БД, seed и запуск HTTP UI.
- `app/service.py` — прикладной сервис для dashboard, списка проектов и карточки проекта.
- `app/models.py` — модели проектов, документов, связей, нехваток и записи в журнале задач.
- `db/setup.py` — SQL схема таблиц `projects`, `project_documents`, `project_links`, `project_deficits`, `module_events`, `task_registry_log`.
- `db/repository.py` — доступ к SQLite, seed демонстрационных данных и агрегаты по проектам.
- `ui/server.py` — HTTP сервер и маршруты `/`, `/projects`, `/projects/<id>`.
- `ui/templates.py` — HTML страницы статуса, списка проектов и карточки проекта.
- `data/project_builder.sqlite3` — собственная БД модуля, создается при первом запуске.
- `logs/project_builder.log` — локальный лог запуска и ключевых действий.

## UI
Доступны страницы:
- `/` — статус модуля, счетчики, краткий список проектов и локальный журнал реестра задач.
- `/projects` — список всех проектов.
- `/projects/<id>` — карточка проекта.

На карточке проекта показываются:
- список документов проекта;
- объяснение, почему документ отнесен к проекту;
- найденные связи между документами;
- список нехваток с критичностью;
- агрегированная полнота проекта.

## База данных
У модуля своя отдельная SQLite БД:

```text
data/project_builder.sqlite3
```

Таблицы:
- `projects` — проекты и общий статус полноты.
- `project_documents` — документы проекта и причина сопоставления.
- `project_links` — найденные связи между документами.
- `project_deficits` — отсутствующие обязательные материалы.
- `module_events` — журнал ключевых событий модуля.
- `task_registry_log` — локальный журнал записей в формате SQL-реестра задач.

## Основные сценарии
1. Запустить модуль через `./modulectl start project_builder`.
2. Открыть UI на `http://127.0.0.1:8103/`.
3. Посмотреть список проектов и общие счетчики.
4. Открыть карточку проекта и увидеть документы, связи и нехватки.
5. Проверить, что демонстрационные данные и стартовая запись в журнале задач сохранены в собственной БД.

## Что подготовлено в стартовой реализации
- Есть entrypoint через `main.py`.
- Есть разделение на `app`, `db`, `ui`.
- Есть собственная SQLite БД модуля.
- Есть seeded демонстрационные проекты без импорта и OCR.
- Есть локальный UI со списком проектов, связями и нехватками.
- Есть локальный лог и таблица `task_registry_log` для фиксации действий в формате реестра задач.

## Ограничения стартовой версии
- Данные пока только демонстрационные и seeded при первом запуске.
- Полнота проекта считается по простому правилу: заполненные документы против открытых дефицитов.
- Централизованная фиксация задач должна идти через общий `./task add ... --sync`; локальная таблица `task_registry_log` используется только как внутренний журнал модуля.
