# orchestrator

## Роль модуля
`orchestrator` отвечает только за координацию: маршрутизацию задач между process-модулями и reviewer-агентами, фиксацию handoff и наблюдение за статусами. Этот модуль не реализует бизнес-логику других сервисов, не заменяет `shell` и не становится общим монолитом.

Каркас в этой папке нужен для явной фиксации границ:
- orchestrator управляет потоком задач;
- orchestrator не исполняет предметную логику других модулей;
- orchestrator не поднимает shell-фронт из этой папки.

## Что делает orchestrator
- читает SQL workflow-базу задач как источник правды перед маршрутизацией;
- определяет, какому `target_agent` передать следующий шаг;
- фиксирует handoff, blocker, review и завершение этапов новыми записями;
- поддерживает единые правила интеграции между модулями.

## Что orchestrator не делает
- не пишет бизнес-логику process-модулей;
- не хранит чужие предметные данные у себя;
- не реализует shell-фронт в этой папке;
- не подменяет собой локальные интерфейсы модулей.

## Статусы и типы действий
### Action types
- `start` — начало работы по задаче;
- `progress` — промежуточное состояние или уточнение;
- `handoff` — передача следующему агенту или модулю;
- `review` — результат проверки reviewer-агента;
- `done` — завершение работы;
- `blocked` — обнаружен блокер;
- `error` — ошибка выполнения.

### Status values
- `todo` — задача зарегистрирована, но не взята в работу;
- `in_progress` — работа идет;
- `ready_for_review` — модульный этап завершен и передан reviewer-агенту;
- `reviewed` — этап проверен reviewer-агентом;
- `done` — этап полностью завершен после обязательных проверок;
- `blocked` — работа остановлена из-за блокера;
- `failed` — работа завершилась ошибкой.

## Как происходит handoff
1. `orchestrator` читает свою очередь или карточку задачи через `../task mine ...` и `../task show ...`.
2. По результату маршрутизации создается новый handoff или событие в workflow-базе через `../task add ... --source-agent orchestrator --sync`.
3. В записи handoff указываются `task_id`, `module_name`, `target_agent`, краткий `summary`, `implementation_report`, `checks_required` и связанные артефакты.
4. История движения задачи остается в `task_events`, а текущий статус задачи агрегируется task-системой в `tasks`.

Шаблон handoff:

```bash
../task add \
  --task-id <task_id> \
  --source-agent orchestrator \
  --target-agent <next_agent> \
  --module-name <module_name> \
  --action-type handoff \
  --status pending \
  --summary "<кому и что передано>" \
  --implementation-report "<какой контекст и где проверять>" \
  --checks-required "<что должен сделать следующий агент>" \
  --artifacts "<paths>" \
  --sync
```

## Источник правды и дашборд
- SQL workflow-база задач — единственный источник правды по состоянию задач.
- Google Sheets используется только как пользовательский дашборд наблюдения.
- Запись в Google Sheets выполняется не вручную, а только через `../task add ... --sync`.
- Если между SQL и дашбордом есть расхождение, приоритет всегда у SQL workflow-базы.

Практически это означает:
1. Перед маршрутизацией orchestrator читает SQL workflow-базу.
2. Решение о следующем шаге фиксируется новой записью в SQL.
3. Google Sheets получает только зеркальную запись для наблюдения пользователем.

## Структура папки
- `main.py` — entrypoint модуля; выводит краткое описание роли и границ orchestrator.
- `app/bootstrap.py` — минимальная сборка профиля orchestrator.
- `app/orchestrator.py` — структура данных с ролью, обязанностями и ограничениями.
- `app/statuses.py` — используемые action types и status values.
- `AGENTS.md` — локальная политика orchestrator.
- `README.md` — описание роли, структуры и правил handoff.
- `logs/` — локальные логи модуля при необходимости.
- `tests/` — место для локальных тестов orchestrator.

## Запуск
Минимальный запуск entrypoint:

```bash
python3 main.py
```

`main.py` не поднимает shell-интерфейс и не запускает бизнес-процессы других модулей. Он только выводит краткую сводку о роли orchestrator, используемых статусах и границах ответственности.
