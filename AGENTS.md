# Lapushka 5 — общая политика для всех агентов

## Цель проекта
Проект строится как модульная система из автономных сервисов. Каждый сервис разрабатывается и запускается отдельно, имеет собственную базу данных, собственный UI на отдельном порту, собственный README и собственную локальную политику агента.

## Главные принципы
1. Не ломать модульность.
2. Не создавать скрытые зависимости между модулями.
3. Не менять чужие папки без явного разрешения.
4. Любое изменение должно быть минимально достаточным.
5. Сначала рабочая простая версия, потом улучшения.
6. Любая логика должна быть понятна по структуре файлов.
7. Любой важный файл должен быть отражен в README модуля.
8. Любая внешняя интеграция должна иметь защиту от ошибок и таймаутов.
9. Любой UI должен быть простым, рабочим и легким.
10. Любая база данных должна быть проверяема и прозрачна по структуре.

## Общие ограничения
- Не переименовывать корневые папки проекта.
- Не переносить файлы между модулями без необходимости.
- Не добавлять тяжелые зависимости без причины.
- Не делать из одного модуля центр управления другим модулем.
- Не писать код "на будущее", если это не требуется текущей задачей.
- Не вносить массовый рефакторинг без отдельной задачи.
- Не хранить секреты в коде, использовать только `.env`.
- Не менять `.env` без явного указания.
- Не удалять существующие файлы без крайней необходимости.
- Не изменять записи в общей SQL-таблице задач; разрешено только добавление новых записей.

## Политика по коду
- Код должен быть простым, читаемым, разбитым по ответственности.
- Каждый модуль обязан иметь `main.py` как точку входа.
- Бизнес-логика не должна лежать в UI-слое.
- Работа с БД должна быть вынесена отдельно.
- Работа с внешними API должна быть вынесена отдельно.
- Конфигурация должна быть вынесена отдельно.
- Любой нетривиальный участок кода должен быть понятен без гадания.
- Предпочтительны короткие функции и явные имена.
- Любой новый файл должен быть отражен в `README.md` модуля.

## Политика по UI
- UI каждого модуля должен быть минимальным и рабочим.
- UI должен показывать только полезные действия и результат.
- UI не должен быть декоративным.
- Нельзя перегружать страницу тяжелыми элементами.
- Все модули должны визуально придерживаться общего стиля, который задает shell-модуль.

## Политика по БД
- У каждого модульного процессника своя БД.
- Нельзя читать или изменять БД другого процессника напрямую.
- Все изменения схемы должны быть предсказуемыми.
- Нужны базовые проверки целостности данных.
- Нужна возможность быстро понять, какие таблицы за что отвечают.

## Политика по логированию
- Логировать старт, завершение, ошибки, ключевые этапы.
- Не засорять лог бессмысленным шумом.
- Логи должны помогать понять, что сломалось и на каком шаге.

## Политика по README
Каждый модуль обязан вести свой `README.md`, где указано:
- назначение модуля;
- структура папки;
- в каких файлах что находится;
- как запускать;
- какой порт используется;
- где лежит БД;
- какие есть основные сценарии работы.

## Политика по задачам
- Источник правды по задачам — SQL-таблица.
- Google Sheets используется как дашборд для наблюдения пользователем.
- Агент может только добавлять новую запись о своем действии, результате или замечании.
- Агент не меняет и не удаляет старые записи.
- Любая проверка, замечание или handoff должны фиксироваться новой записью.

## Политика по безопасности
- Не выводить секреты в логи.
- Не хранить ключи в коде.
- Не отправлять наружу лишние данные.
- Любой внешний вызов должен быть ограничен по таймауту и ошибкам.

## Политика принятия решений
При споре между "быстрее" и "проще поддерживать" выбирать вариант, который проще поддерживать, если он не ломает сроки критически.

## Политика завершения работы
Задача считается завершенной только если:
- код работает;
- структура не сломана;
- README обновлен;
- есть запись в реестре задач;
- модуль запускается;
- результат можно проверить.
