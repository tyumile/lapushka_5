# Cabinets Plan

## Цель
Перевести текущую структуру проекта из режима одного кабинета в режим выбора между несколькими кабинетами без поломки модульности.

## Базовый принцип
Кабинет должен стать явным системным контекстом. Все пользовательские сценарии, данные модулей, task workflow и reviewer-проверки должны быть привязаны к `cabinet_id`.

## План

### 1. Зафиксировать модель кабинета как общий контракт
- создать минимальное описание сущности `cabinet`;
- завести поля: `id`, `slug`, `title`, `status`, `created_at`;
- зафиксировать правило, что все пользовательские сценарии process-модулей выполняются в контексте выбранного `cabinet_id`.

### 2. Ввести кабинет в shell как системный контекст
- сделать экран выбора кабинета в shell;
- перевести верхние маршруты на вид `/cabinet/<cabinet_id>/...` или эквивалентный query-параметр;
- shell должен хранить текущий кабинет только как навигационный контекст, а не как источник бизнес-данных.

### 3. Добавить `cabinet_id` в task workflow
- расширить `tasks`, `task_handoffs`, `task_reviews`, `task_events` полем `cabinet_id`;
- обновить `./task`, чтобы `mine`, `show`, `list`, `add` умели работать с кабинетом;
- привязать handoff, review и статусы к конкретному кабинету.

### 4. Добавить `cabinet_id` в БД каждого process-модуля
- `ingest_registry`: добавить `cabinet_id` в основные таблицы и индексы;
- `doc_classifier`: добавить `cabinet_id` в таблицы документов, распознавания и классификации;
- `project_builder`: добавить `cabinet_id` в таблицы проектов, документов, связей и дефицитов;
- каждая модульная БД остается своей, но внутри становится многокабинетной.

### 5. Протянуть `cabinet_id` через repository, service и UI каждого модуля
- все list/get/create/update операции должны явно принимать `cabinet_id`;
- UI каждого модуля должен работать только в рамках выбранного кабинета;
- прямые запросы без `cabinet_id` должны считаться legacy-режимом только на этапе миграции.

### 6. Сделать миграцию текущего одиночного кабинета
- создать системный кабинет `default`;
- присвоить ему все существующие данные;
- сохранить обратную совместимость без потери текущего состояния.

### 7. Обновить shell-маршруты до многокабинетной навигации
- разделы `Источники`, `Рабочая документация`, `Проект` должны открываться уже не как глобальные страницы модуля, а как страницы выбранного кабинета;
- shell должен уметь переключать кабинет без ручного редактирования URL пользователем.

### 8. Обновить правила handoff и review
- модульные агенты обязаны писать `cabinet_id` в task workflow;
- в `implementation_report` указывать, какой кабинет проверять;
- reviewer-агенты должны брать задачи и handoff только для конкретного кабинета.

### 9. Обновить документацию и политики агентов
- обновить `AGENTS.md`, `shared/task_registry.md`, `README.md` и локальные `AGENTS.md` модулей;
- явно зафиксировать, что система многокабинетная и все сценарии выполняются в контексте кабинета.

### 10. Внедрять поэтапно
- сначала `shared + shell + task workflow`;
- затем `ingest_registry`;
- затем `doc_classifier`;
- затем `project_builder`;
- после этого включать полноценный выбор кабинета в пользовательском потоке.

## Практический порядок внедрения
1. `shared`: описать контракт кабинета.
2. `shell`: выбор кабинета и маршрутизация.
3. `task workflow`: `cabinet_id` в новой базе.
4. `ingest_registry`: первый process-модуль с реальным `cabinet_id`.
5. `doc_classifier`.
6. `project_builder`.
7. Документация и правила агентов.
8. Миграция старых данных в `default` cabinet.
9. Проверка сценария: выбрать кабинет -> зайти в модуль -> создать данные -> передать на review -> проверить в том же кабинете.

## Короткий итог
Сейчас система однокабинетная потому, что кабинет не является первичной сущностью данных. Для поддержки нескольких кабинетов нужно сделать `cabinet_id` обязательным контекстом на уровне shell, API, БД модулей и task workflow.
