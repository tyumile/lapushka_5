# backend_reviewer

## Назначение
`backend_reviewer` — отдельный reviewer-модуль для проверки завершенных бэкенд-процессов без правки их кода.

Он нужен, чтобы:
- читать завершенные изменения других модулей;
- проверять структуру, точку входа и базовый запуск;
- фиксировать короткие проверяемые рекомендации;
- добавлять результат review новой записью в реестр задач.

## Ограничения роли
- не изменяет чужой код;
- не лезет в чужие БД;
- не запускает тяжелые сценарии;
- не становится оркестратором;
- работает только как независимый reviewer-модуль.

## Структура папки
- `main.py` — минимальный entrypoint, который подтверждает готовность reviewer-модуля.
- `AGENTS.md` — локальная политика reviewer-а.
- `README.md` — описание роли, структуры и процесса review.
- `app/__init__.py` — пакет reviewer-модуля.
- `app/review_workflow.py` — минимальная локальная логика для описания роли и путей шаблонов.
- `prompts/recommendations/recommendation_template.md` — каркас рекомендаций по backend review.
- `prompts/reports/review_report_template.md` — каркас итоговой записи в реестр задач.
- `tests/README.md` — место под локальные smoke-проверки reviewer-модуля.
- `logs/` — локальные логи reviewer-модуля при необходимости.

## Процесс review
1. Reviewer сначала читает свою очередь через `../../task mine --agent backend_reviewer ...`, а затем при необходимости раскрывает историю через `../../task list ...`.
2. Reviewer смотрит изменения целевого модуля, не меняя его файлы.
3. Reviewer проверяет структуру, роль `main.py`, разделение ответственности и риски запуска.
4. Reviewer формирует рекомендации по шаблону в `prompts/recommendations/`.
5. Reviewer фиксирует результат через `../../task add ... --action-type review --sync` по шаблону в `prompts/reports/`.

## Запуск
Минимальная проверка entrypoint:

```bash
python main.py
```

Ожидаемый результат:
- модуль печатает свою роль;
- показывает пути к шаблонам рекомендаций и отчетов;
- не пытается запускать чужие сервисы и не обращается к чужим БД.

## Порт и БД
- Порт не используется, потому что это reviewer-модуль без UI и без собственного сервера.
- Собственная БД не используется.

## Основные сценарии
- подготовить review завершенного backend-модуля;
- сформировать короткие рекомендации для основного агента;
- записать итог проверки в `task_reviews` и синхронизировать dashboard.
